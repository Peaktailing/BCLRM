## 一、项目概述
### 1.1 项目目标
基于 Teable 低代码数据库 + Streamlit 前端，开发一套轻量级试剂库存管理系统，实现试剂的**入库、领用、归还、查询、统计**全流程管理，支持管控化学品特殊标记，预留耗材管理扩展接口。

### 1.2 技术栈
| 层级 | 技术 | 版本要求 |
|------|------|----------|
| 前端 | Streamlit | ≥1.40.0 |
| 后端 | Python | ≥3.8 |
| 数据库 | Teable | 本地部署版 |
| 依赖 | requests, pandas | 最新稳定版 |

## 二、整体架构与分层职责
项目采用**五层架构**，严格遵循「单一职责原则」，各层之间只能向下调用，不能反向依赖。

```
┌─────────────────────────────────┐
│ 页面层 (pages/)                 │  只负责UI渲染和用户交互，无业务逻辑
├─────────────────────────────────┤
│ 业务逻辑层 (business/)          │  核心！所有业务规则、数据校验、流程控制
├─────────────────────────────────┤
│ API服务层 (services/)           │  只负责与Teable数据库通信，单表CRUD
├─────────────────────────────────┤
│ 数据模型层 (models/)            │  纯数据结构，与Teable表一一对应
├─────────────────────────────────┤
│ 配置与工具层 (config/ + utils/) │  全局配置、通用工具函数
└─────────────────────────────────┘
```

### 各层职责说明
1. **页面层**：只写 Streamlit 组件（输入框、按钮、表格），调用业务层接口获取数据，展示结果。**绝对不能直接调用 Teable API 或写业务逻辑**。
2. **业务逻辑层**：所有业务规则（如“领用数量不能超过剩余量”“管控试剂需要审批”）都在这里实现，与UI完全解耦，方便单元测试。
3. **API服务层**：封装 Teable 的 HTTP 请求，每个表对应一个 Service 类，只做单表的增删改查，不处理任何业务逻辑。
4. **数据模型层**：用 `dataclass` 定义与 Teable 表完全一致的数据结构，字段名、类型严格匹配 API 文档。
5. **配置与工具层**：所有全局配置、常量、通用工具函数（日期处理、编号生成）统一放在这里，避免硬编码。
6. **全项目统一编码范式**：优先使用面向对象（Class + dataclass）开发，严控面向过程脚本式写法
models 层强制 dataclass 数据实体类，禁止零散字典传参；
services、business 层统一封装业务类，将关联属性、复用方法收拢到类内，不写大量全局函数、全局变量；
utils、components 通用能力封装为工具类 / 组件类，避免堆砌无归属独立函数；
仅极简一次性工具脚本允许面向过程，业务核心代码一律禁止纯面向过程实现。

## 四、开发规范（必须严格遵守）
### 4.1 命名规范
| 类型 | 规范 | 示例 |
|------|------|------|
| 文件夹 | 小写+下划线 | `reagent_bottle`, `storage_requirement` |
| 文件 | 小写+下划线 | `inventory_service.py`, `1_实时库存.py` |
| 类名 | 大驼峰 | `ReagentBottle`, `BaseTeableService` |
| 函数/方法 | 小写+下划线 | `get_reagent_by_barcode`, `create_borrow_record` |
| 变量 | 小写+下划线 | `bottle_number`, `remaining_quantity` |
| 常量 | 全大写+下划线 | `TEABLE_BASE_URL`, `DEFAULT_UNIT` |

### 4.2 代码规范
1. **缩进**：使用4个空格，禁止使用Tab
2. **注释**：
   - **所有功能块必须先写注释**：类、函数、方法必须先写文档字符串（docstring），说明功能、参数、返回值
   - 复杂业务逻辑必须加行内注释
   - 临时注释必须标注日期和作者
3. **错误处理**：
   - 所有API请求必须加异常捕获
   - 业务逻辑错误必须返回明确的提示信息
   - 禁止使用裸`except`，必须捕获具体异常
4. **硬编码**：
   - 所有字符串常量、数字常量必须定义在`config/settings.py`中
   - 所有字段名必须定义在`utils/field_mapper.py`中，禁止直接写字符串
5. **文档更新**：
   - **每次更新代码必须同步更新开发文档**（docs/开发基础文档.md）和对应的使用说明（docs/用户使用手册.md）
6. **代码规范**：
【编码范式强制要求】
项目所有核心层（models /services/business /components）必须采用面向对象编程，使用类封装数据与行为；
禁止将业务逻辑、数据库操作、数据实体拆分为大量独立全局函数；
数据传输统一使用 dataclass 实体对象，不裸传 dict；
仅 utils 内一次性简单辅助函数可使用面向过程写法，业务流程、数据库交互全部封装类；
禁止在页面层编写大量过程式判断、循环、数据计算，复杂逻辑封装至 business 类内调用。

### 4.3 数据操作规范
1. **管控化学品数据表**：只能读取，不能修改。该表为系统参考数据，用于判断试剂是否为管控试剂，测试后必须删除对应的测试数据。
2. **试剂瓶信息表**：核心业务表，所有操作需经过业务逻辑层校验。
3. **领用/归还记录表**：自动生成，禁止手动修改。

### 4.4 提交规范
每次提交代码必须遵循以下格式：
```
<类型>: <描述>
```
类型说明：
- `feat`: 新增功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整（不影响功能）
- `refactor`: 代码重构
- `test`: 测试相关
- `chore`: 构建工具或依赖更新

示例：
```
feat: 实现试剂入库核心业务逻辑
fix: 修复领用数量超过剩余量的bug
docs: 更新开发基础文档，添加功能对照表
```

### 4.5 页面层与业务层 职责分离规范（永久强制）

#### 4.5.1 页面层（pages/） 绝对职责
1. 只负责 UI 渲染：输入框、下拉框、单选框、复选框、按钮、表格、表单布局
2. 只负责数据展示：列表、详情、状态、提示语（成功/失败/警告）
3. 只负责调用 business/ 层的函数，不编写任何业务逻辑
4. 禁止编写：编号生成、数据校验、状态判断、附件处理、API 直接调用
5. 下拉选择框数据：必须从 services/ 层获取，禁止硬编码
6. 表单提交：仅收集参数 → 调用业务函数 → 展示结果

#### 4.5.2 业务层（business/） 绝对职责
1. 所有业务规则唯一实现地：编号生成、数据校验、关联匹配、状态计算
2. 所有数据处理：附件解析、格式转换、模糊查询、自动匹配
3. 统一调用 services/ 层完成数据库 CRUD
4. 统一返回标准格式：(bool: 是否成功, str: 提示信息 / data: 数据)
5. 禁止包含任何 Streamlit 前端代码（st.xxx 语法）
6. 禁止直接操作页面，只处理数据与逻辑
7.强制面向对象约束。所有业务能力封装为业务 Service 类，相关操作作为类方法，禁止分散全局过程函数；业务所需数据统一传入 dataclass 模型对象，不裸字典交互。
#### 4.5.3 关联表下拉框开发规范
1. 试剂类型 → 从 reagent_type 表获取
2. 存储要求 → 从 storage_requirement 表获取
3. 供应商 → 从 supplier 表获取
4. 存储位置 → 从 storage_location 表获取
5. 人员 → 从 person 表获取
6. 所有下拉框必须动态加载，禁止手写固定选项；读取数据后统一转换为 model 实体对象，页面仅读取对象属性展示，避免过程式字典取值。

#### 4.5.4 搜索功能规范
1. 关键字搜索默认使用**模糊匹配**
2. 多字段搜索：同时匹配主名称 + 显示名称
3. 搜索不区分大小写
4. 搜索逻辑必须写在 business/ 层，页面只传关键词

#### 4.5.5 附件上传规范
1. MSDS 等附件：页面只负责上传文件
2. 业务层负责处理文件、存储路径/内容
3. 禁止在页面层处理文件逻辑

## 六、常见问题定位指南
### 6.1 页面打不开
1. 检查Streamlit是否启动：看终端是否有`You can now view your Streamlit app`提示
2. 检查端口是否被占用：运行`ss -tuln | grep 端口号`，如果被占用，换端口启动
3. 检查防火墙：运行`sudo ufw allow 端口号/tcp`放行端口
4. 检查访问地址：必须用服务器真实IP，不能用`0.0.0.0`

### 6.2 页面能打开，但没有数据
1. 检查Teable服务是否运行：浏览器访问`http://172.22.132.59:3000`
2. 检查Token是否正确：确认`config/settings.py`中的`TEABLE_API_TOKEN`
3. 检查表ID是否正确：确认`config/settings.py`中的`TABLE_IDS`
4. 检查API请求日志：看终端是否有`Teable API错误`提示

### 6.3 操作失败（入库/领用/归还没反应）
1. 检查必填字段是否都填了
2. 检查字段类型是否正确（如数字字段不能填字符串）
3. 检查业务逻辑限制（如领用数量不能超过剩余量）
4. 看终端报错信息，定位到对应的业务层文件

### 6.4 代码修改后页面没变化
1. 刷新浏览器页面（按F5）
2. 重启Streamlit服务（按Ctrl+C，再重新启动）
