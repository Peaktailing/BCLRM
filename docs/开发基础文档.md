一、项目概述
1.1 项目目标
基于 SQLite 数据库（原生 sqlite3 模块）+ Streamlit 前端，开发一套轻量级试剂库存管理系统，实现试剂的入库、领用、归还、查询、统计全流程管理，支持管控化学品特殊标记，预留耗材管理扩展接口。

二、整体架构与分层职责
项目采用六层架构，严格遵循「单一职责原则」，各层之间只能向下调用，不能反向依赖。
┌─────────────────────────────────┐
│ 页面层 (pages/)                 │  只负责UI渲染和用户交互，无业务逻辑
├─────────────────────────────────┤
│ 业务逻辑层 (business/)          │  核心！所有业务规则、数据校验、流程控制
├─────────────────────────────────┤
│ ORM 服务层 (services/)          │  封装数据库操作，单表标准化CRUD
├─────────────────────────────────┤
│ 数据库访问层 (db/)              │  数据库连接管理、通用CRUD基类
├─────────────────────────────────┤
│ 数据模型层 (models/)            │  数据表对应的 dataclass 实体类
├─────────────────────────────────┤
│ 配置与工具层 (config/ + utils/) │  全局配置、通用工具函数
└─────────────────────────────────┘
各层职责完整说明
1. 页面层：只写 Streamlit 组件（输入框、按钮、表格、弹窗、筛选控件），仅调用业务层类方法获取数据并渲染展示。永久禁止直接操作数据库、编写业务校验逻辑、拼接SQL语句。
2. 业务逻辑层：系统核心层，全部业务规则统一在此实现，例如领用数量校验、管控化学品审批、库存自动更新、多表关联统计等；与前端UI完全解耦，仅依赖ORM服务层完成数据读写，便于独立单元测试。所有业务方法统一返回 ServiceResult 对象。
3. ORM 服务层：基于 db/base_service.py 继承扩展，每张数据表对应独立 Service 类，仅负责单表增删改查、基础查询，不承载任何业务判断、流程规则。
4. 数据库访问层：封装 SQLite 数据库连接管理、表结构初始化、基础 CRUD 方法（execute_query、execute_insert、execute_update 等），提供 BaseService 通用 CRUD 基类供 services 层继承。
5. 数据模型层：使用 Python dataclass 定义数据实体类，每个类与 SQLite 数据表一一对应，字段、数据类型在类内统一声明；全项目数据流转统一使用实体对象传递，禁止裸字典传输数据。
6. 配置与工具层：全局数据库连接参数、系统常量、编号生成、日期转换、统一异常处理等通用能力统一存放，所有硬编码常量、字段标识全部移入此处，业务代码禁止手写固定字符串、数字。
全项目统一编码范式（强制规范）
项目所有核心代码优先使用面向对象Class开发，严格限制面向过程脚本式写法，分层强制约束如下：
1. models层：全部为 dataclass 实体类，数据表结构与类定义强绑定，禁止零散字典、元组承载表数据；
2. services、business层：全部能力封装为独立Class，关联属性、复用读写/校验逻辑收拢为类内方法，不批量创建无归属全局函数、全局变量；
3. utils、components通用层：通用工具、复用UI组件统一封装为工具类、组件类，避免堆砌零散独立过程函数；
4. 例外规则：仅一次性简易运行脚本可采用面向过程写法，models/services/business核心业务代码一律禁止纯面向过程实现；
5. 数据库编码约束：通过 db/ 层统一进行数据库读写，禁止在页面层和业务层直接拼接、手写原始 SQL 字符串执行查询/修改操作。

4.2 代码规范
1. 缩进：使用4个空格，禁止使用Tab
2. 注释：
- 所有功能块必须先写注释：类、函数、方法必须先写文档字符串（docstring），说明功能、参数、返回值
- 复杂业务逻辑必须加行内注释
- 临时注释必须标注日期和作者
3. 错误处理：
- 所有数据库ORM请求必须加异常捕获
- 业务逻辑错误必须返回明确的提示信息
- 禁止使用裸except，必须捕获具体异常
4. 硬编码：
- 所有字符串常量、数字常量必须定义在config/settings.py中
- 所有字段名必须定义在utils/field_mapper.py中，禁止直接写字符串
5. 文档更新：
- 每次更新代码必须同步更新开发文档（docs/开发基础文档.md）和对应的使用说明（docs/用户使用手册.md）
6. 编码范式强制要求：
项目所有核心层（models / services / business / components）必须采用面向对象编程，使用类封装数据与行为；禁止将业务逻辑、数据库操作、数据实体拆分为大量独立全局函数；数据传输统一使用 ORM 实体对象，不裸传 dict；仅 utils 内一次性简单辅助函数可使用面向过程写法，业务流程、数据库交互全部封装类；禁止在页面层编写大量过程式判断、循环、数据计算，复杂逻辑封装至 business 类内调用。
4.3 数据操作规范
1. 管控化学品数据表：只能读取，不能修改。该表为系统参考数据，用于判断试剂是否为管控试剂，测试后必须删除对应的测试数据。
2. 试剂瓶信息表：核心业务表，所有操作需经过业务逻辑层校验。
3. 领用/归还记录表：自动生成，禁止手动修改。
4.4 Git 提交规范
所有代码开发完成、审查通过后，由主智能体统一整理、合并、提交代码，严格遵循以下提交格式：
<类型>: <描述>
类型说明：
- feat: 新增功能
- fix: 修复bug
- docs: 文档更新
- style: 代码格式调整（不影响功能）
- refactor: 代码重构
- test: 测试相关
- chore: 构建工具或依赖更新
示例：
feat: 实现试剂入库核心业务逻辑
fix: 修复领用数量超过剩余量的bug
docs: 更新开发基础文档，适配SQLite架构
4.5 页面层与业务层 职责分离规范（永久强制）
4.5.1 页面层（pages/） 绝对职责
1. 只负责 UI 渲染：输入框、下拉框、单选框、复选框、按钮、表格、表单布局
2. 只负责数据展示：列表、详情、状态、提示语（成功/失败/警告）
3. 只负责调用 business/ 层的类方法，不编写任何业务逻辑
4. 禁止编写：编号生成、数据校验、状态判断、附件处理、数据库直接调用
5. 下拉选择框数据：必须从业务层动态获取，禁止硬编码
6. 表单提交：仅收集参数 → 调用业务类方法 → 展示结果
4.5.2 业务层（business/） 绝对职责
1. 所有业务规则唯一实现地：编号生成、数据校验、关联匹配、状态计算
2. 所有数据处理：附件解析、格式转换、模糊查询、自动匹配
3. 统一调用 services/ 层 ORM 方法完成数据库 CRUD
4. 统一返回 ServiceResult 对象（定义在 utils/error_handler.py），包含：
   - success: bool — 是否成功
   - message: str — 提示信息
   - data: Any — 返回数据
   - is_success() / is_failure() — 便捷判断方法
5. 禁止包含任何 Streamlit 前端代码（st.xxx 语法）
6. 禁止直接操作页面，只处理数据与逻辑
7. 强制面向对象约束：所有业务能力封装为业务 Service 类，相关操作作为类方法，禁止分散全局过程函数；业务所需数据统一传入 ORM 实体对象，不裸字典交互。
4.5.3 关联表下拉框开发规范
1. 试剂类型 → 从 reagent_type 表获取
2. 存储要求 → 从 storage_requirement 表获取
3. 供应商 → 从 supplier 表获取
4. 存储位置 → 从 storage_location 表获取
5. 人员 → 从 person 表获取
6. 所有下拉框必须动态加载，禁止手写固定选项；读取数据后统一转换为 ORM model 实体对象，页面仅读取对象属性展示，避免过程式字典取值。
4.5.4 搜索功能规范
1. 关键字搜索默认使用模糊匹配
2. 多字段搜索：同时匹配主名称 + 显示名称
3. 搜索不区分大小写
4. 搜索逻辑必须写在 business/ 层，页面只传关键词
4.5.5 附件上传规范
1. MSDS 等附件：页面只负责上传文件
2. 业务层负责处理文件、存储路径/内容
3. 禁止在页面层处理文件逻辑

五、常见问题定位指南
5.1 页面打不开
1. 检查Streamlit是否启动：看终端是否有You can now view your Streamlit app提示
2. 检查端口是否被占用，若被占用，更换端口重启服务
3. 检查服务器防火墙，放行对应端口
4. 检查访问地址，使用服务器真实IP访问
5.2 页面能打开，但没有数据
1. 检查SQLite数据库文件是否存在、是否损坏
2. 检查数据库初始化是否完成，数据表是否正常创建
3. 检查config/settings.py数据库配置是否正确
4. 查看终端日志，定位ORM查询异常、数据为空等问题
5.3 操作失败（入库/领用/归还没反应）
1. 检查表单必填字段是否完整填写
2. 检查字段传入类型是否与ORM实体类类型匹配
3. 核对业务层校验规则（如领用数量不大于库存数量）
4. 查看终端报错，定位对应业务类、ORM服务类异常
5.4 代码修改后页面没变化
1. 刷新浏览器页面（F5）
2. 重启Streamlit服务
3. 清除Streamlit本地缓存

5.5 错误日志管理
1. 系统运行时产生的所有错误、警告信息统一记录在 `logs/error.log` 文件中
2. 错误日志格式：`时间 | 级别 | 模块 | 错误信息 | 文件位置 | 解决方案`
3. 常见错误级别：
   - ERROR：严重错误，影响系统运行
   - WARNING：警告信息，需要关注但不影响运行
4. 遇到问题时，优先查看 error.log 文件定位问题原因
5. 修复问题后，需在 error.log 中补充解决方案说明
